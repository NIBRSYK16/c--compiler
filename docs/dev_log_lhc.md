## 开发日志（lhc）

### 2026.4.30 - 2026.5.2

主要负责词法分析模块的设计和实现，前期先把实验三里写过的自动机构造思路迁移到大作业项目里，并按照当前项目框架重新整理接口。

已完成：

- 设计了词法分析模块的数据结构，包括 `TokenRule`、`NFAState`、`NFAFragment`、`DFAState` 等，分别用于描述词法规则、NFA 片段和 DFA 状态。
- 实现 Thompson 风格的 NFA 构造辅助类，支持字符集合、字符串字面量、连接、选择、闭包和正闭包等基本构造。
- 按 C-- 语言要求登记关键字、运算符、界符、标识符、整数和浮点数规则。
- 实现 NFA 到 DFA 的子集构造算法，包括 `epsilonClosure`、`moveSet` 和接受规则优先级选择。
- 实现 DFA 最小化，先按接受规则分组，再根据转移签名反复细化分组。
- 初步完成 `Lexer::tokenize(...)`，使用最小化 DFA 进行最长匹配扫描，并输出项目统一的 `LexResult`。

### 2026.5.3 - 2026.5.5

把词法分析器和语法分析器的接口对齐，重点处理 token 字段命名和属性值格式。

已完成：

- 在 `Token` 中区分 `lexeme`、`type`、`grammar`、`attr` 四类信息，避免词法输出和语法终结符混在一起。
- 将标识符、整数、浮点数分别映射为 Parser 使用的 `Ident`、`IntConst`、`floatConst`。
- 对关键字统一输出小写文法符号，使 `INT`、`Int` 这类输入仍能归一化成 `int`。
- 按指导书表格修正关键字、运算符和界符的属性编号。
- 保留最长匹配策略，保证 `>=`、`<=`、`==`、`!=`、`&&`、`||` 不会被拆成更短 token。
- 用常见程序片段测试了声明、赋值、函数定义、`if/else`、`return`、一元 `!` 和浮点常量等 token。

### 2026.5.6

配合阶段测试，把词法分析模块接入临时 driver 和 Makefile。

已完成：

- 和测试 driver 对接，支持从 `.sy` 文件读取源码并输出 `token.tsv`。
- 确认 `make lexer_test` 和 `make run-lexer` 能单独编译运行词法阶段。
- 调整词法错误返回方式，不在模块内部直接退出，而是通过 `LexResult::success` 和 `errorMessage` 交给主流程判断。
- 完善行号、列号维护逻辑，保证空白字符、换行和错误字符都能正确推进位置。

### 2026.5.8

在已有词法分析器基础上继续做增强和边界优化，主要目标是让模块更接近实际编译器前端，减少后续 Parser 收到脏 token 的情况。

已完成：

- 增加 `//` 行注释跳过逻辑，注释内容不再被识别成 `/` 运算符或标识符。
- 增加 `/* ... */` 块注释跳过逻辑，并支持块注释跨多行时正确维护行号和列号。
- 增加未闭合块注释报错，例如文件结束前没有遇到 `*/` 时，会报告注释开始的行列号。
- 增加畸形数字字面量检查，`123abc`、`1.`、`1.a` 这类输入会整体报词法错误，而不是拆成多个看似合法但语义混乱的 token。
- 改进词法错误信息，使非法字符、畸形数字、未闭合注释都能给出更明确的错误类型。
- 补充 `tokenTypeToString` 的内联实现，避免后续其他模块或测试程序使用时出现只有声明没有定义的问题。
- 增加 `AutomataStats` 和 `getAutomataStats()` 接口，用于统计 NFA 状态数、DFA 状态数、最小化 DFA 状态数、规则数和字符表大小。
- 在 `lexer_main` 中增加 `LEXER_STATS` 环境变量开关，方便测试时输出自动机构造规模。
- 新增 `tests/samples/lexer_edge_lhc.sy`，覆盖注释、畸形数字、复合运算符等词法边界情况。
- 更新 `docs/lexer_design.md`，补充注释处理、畸形数字处理和自动机统计接口说明。

### 2026.5.8 晚

继续把词法分析模块往“可诊断、可集成、可展示”的方向升级，重点不是单纯识别 token，而是让错误信息可以被后续阶段和测试报告复用。

已完成：

- 新增公共诊断结构 `Diagnostic`、`DiagnosticSeverity`、`SourceRange`，词法错误不再只有一段普通字符串。
- 在 `LexResult` 中增加 `diagnostics` 字段，保留原有 `errorMessage` 的同时提供结构化错误列表。
- 为词法错误分配稳定错误码，例如 `LEX_UNEXPECTED_CHAR`、`LEX_MALFORMED_NUMBER`、`LEX_UNTERMINATED_COMMENT`。
- 每条词法诊断都记录严重等级、行号、列号、影响范围和修复提示，方便后续生成更清晰的 `error.txt`。
- 将非法字符、畸形数字、未闭合块注释统一接入结构化诊断渲染函数，减少重复拼接错误字符串。
- 增加 JSON Lines 诊断渲染能力，`DIAGNOSTICS_JSON=1` 时可以输出机器可读错误，方便自动化测试或报告脚本使用。
- 更新词法设计文档，补充结构化诊断的设计原因、字段含义和错误码表。
- 保持对原有测试 driver 的兼容，旧流程仍然可以读取 `errorMessage`，新流程可以直接消费 `diagnostics`。

这部分主要提升词法分析模块的工程完整性。后续如果主流程需要输出更像编译器的错误报告，可以直接基于 `Diagnostic` 做统一格式化，不需要改扫描器核心逻辑。
